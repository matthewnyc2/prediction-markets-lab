§1What this program is, plainly

Prediction markets are sites where people buy and sell contracts that pay $1 if a specified event happens (e.g. "Will the Fed cut rates in June?"). The current price of a contract reflects the crowd's probability estimate.

Paper trading means simulating orders — the app pretends you bought the contract, records what would have happened, and tracks a fake portfolio balance. No real money moves.

Backtesting means running a strategy against historical market data to see how it would have performed, so you can evaluate an idea before trading it live.

Why this app exists (from your why.txt): you want to develop and validate quantitative strategies — Kelly sizing, arbitrage, momentum, fade — without financial exposure, and you need performance metrics (P&L, Brier, Sharpe, win rate) to know whether a strategy actually works before risking real capital.

§2Key concepts (so the screens make sense)

Brier score — measures how well-calibrated a probability forecast is. Lower is better; 0 is perfect, 0.25 is a coin-flip guesser. Used here because prediction-market prices are probability forecasts.
Sharpe ratio — risk-adjusted return. (average return − risk-free rate) / volatility. >1 is good, >2 is great. Industry-standard for comparing strategies.
Win rate — % of trades that closed profitable. Useful but incomplete: you can have a low win rate and still make money if your wins are bigger than your losses.
Kelly sizing — a formula for how much of your bankroll to bet when you believe the true probability differs from the market price. Maximises long-run growth.
Arbitrage scanner — watches for the same (or very similar) contract priced differently across the 4 platforms. Buys low on one, sells high on the other, locks in a theoretical risk-free edge.
Market maker — a strategy that quotes both a buy and sell price on a market and profits from the spread, rather than from predicting direction.

§3How the pieces fit together

01 · Ingest
Platform adapters
Pull historical + live markets from Kalshi, Polymarket, Manifold, PredictIt. Normalize into a common format.
02 · Store
Local database
SQLite/Postgres holds markets, prices, trades, fills, portfolio snapshots. No cloud.
03 · Decide
Strategy engine
5 pluggable strategies read market data and emit orders. User toggles which are active.
04 · Simulate
Paper trading engine
Executes orders against recorded price data or live feeds. Updates portfolio.
05 · Evaluate
Metrics & dashboard
Computes P&L, Brier, Sharpe, win rate. Backtester replays history. UI visualises everything.

§6Proposed tech direction (you'll confirm at HACF Step 11)

Frontend

Next.js + TypeScript + Tailwind for the web UI. Next.js because it runs locally with one command, has built-in API routes (so frontend and backend can share a process), and renders quickly enough for a data-dense trading UI. Tailwind keeps styling consistent.

Recharts or ECharts for P&L curves, price history, comparison overlays.

Backend + data

Python FastAPI for the adapters, strategy engine, backtester. Python because the quant ecosystem (pandas, numpy, scipy) is unmatched for backtesting. FastAPI because async is native, OpenAPI docs are automatic.

SQLite → Postgres: start with SQLite for zero-config local dev, upgrade to Postgres if data volume demands it. Schema: markets, prices, trades, fills, portfolio snapshots.

Strategy plugin interface

Each strategy is a Python class with two methods: decide(market_state) → orders and on_fill(fill) → None. Dropping a new file into strategies/ registers it. This makes the 5 built-ins extensible with a 6th, 7th, etc.

Backtest engine

Event-driven replay: reconstruct order book state at each historical tick, feed to strategy, simulate fill with realistic slippage. Same code path as live paper trading — swap the data source. This guarantees backtest results match what live trading would have done.
