# Alembic — Tech KB

**Version pinning**: `alembic@1.13.x` (pairs with SQLAlchemy 2.x). Works with sync and async engines.

## 1. API surface overview

- `alembic init [dir]` — scaffold migration folder
- `alembic revision --autogenerate -m "msg"` — generate migration from model diff
- `alembic upgrade head` / `alembic downgrade -1`
- `alembic history`, `alembic current`, `alembic show <rev>`
- `env.py` — migration environment (customize for async)
- `versions/` — one migration file per revision

## 2. Setup for async SQLAlchemy

### `alembic.ini` (relevant bits)

```ini
[alembic]
script_location = backend/alembic
prepend_sys_path = .
sqlalchemy.url = sqlite+aiosqlite:///./local.db

[loggers]
keys = root,sqlalchemy,alembic
```

### `backend/alembic/env.py` (async-aware)

```python
import asyncio
from logging.config import fileConfig
from sqlalchemy.ext.asyncio import async_engine_from_config
from sqlalchemy import pool
from alembic import context

from backend.app.models import Base  # target metadata

config = context.config
if config.config_file_name:
    fileConfig(config.config_file_name)
target_metadata = Base.metadata

def run_migrations_offline() -> None:
    url = config.get_main_option("sqlalchemy.url")
    context.configure(url=url, target_metadata=target_metadata, literal_binds=True, dialect_opts={"paramstyle": "named"})
    with context.begin_transaction():
        context.run_migrations()

def do_run_migrations(connection) -> None:
    context.configure(connection=connection, target_metadata=target_metadata, render_as_batch=True)
    with context.begin_transaction():
        context.run_migrations()

async def run_migrations_online() -> None:
    connectable = async_engine_from_config(config.get_section(config.config_ini_section, {}), prefix="sqlalchemy.", poolclass=pool.NullPool)
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()

if context.is_offline_mode():
    run_migrations_offline()
else:
    asyncio.run(run_migrations_online())
```

> `render_as_batch=True` is essential for SQLite — it rewrites tables for column drops/alters.

## 3. Code examples

### Example 1 — Initial migration (autogenerated)

```python
# backend/alembic/versions/0001_initial.py
"""initial schema"""
from alembic import op
import sqlalchemy as sa

revision = "0001"
down_revision = None

def upgrade() -> None:
    op.create_table(
        "markets",
        sa.Column("id", sa.String(), primary_key=True),
        sa.Column("platform", sa.String(20), index=True),
        sa.Column("title", sa.String(500)),
        sa.Column("yes_price", sa.Float()),
        sa.Column("no_price",  sa.Float()),
        sa.Column("volume_24h", sa.Float(), server_default="0"),
        sa.Column("closes_at", sa.DateTime(), nullable=True),
        sa.Column("created_at", sa.DateTime(), server_default=sa.func.now()),
    )
    op.create_table(
        "price_ticks",
        sa.Column("id", sa.Integer(), primary_key=True, autoincrement=True),
        sa.Column("market_id", sa.String(), sa.ForeignKey("markets.id", ondelete="CASCADE"), index=True),
        sa.Column("ts", sa.DateTime(), index=True),
        sa.Column("yes", sa.Float()),
        sa.Column("no",  sa.Float()),
    )
    op.create_index("ix_market_ts", "price_ticks", ["market_id", "ts"])

def downgrade() -> None:
    op.drop_index("ix_market_ts", table_name="price_ticks")
    op.drop_table("price_ticks")
    op.drop_table("markets")
```

### Example 2 — Adding a column (SQLite-safe batch)

```python
def upgrade() -> None:
    with op.batch_alter_table("markets") as batch:
        batch.add_column(sa.Column("liquidity_usd", sa.Float(), server_default="0"))

def downgrade() -> None:
    with op.batch_alter_table("markets") as batch:
        batch.drop_column("liquidity_usd")
```

### Example 3 — Data migration

```python
def upgrade() -> None:
    op.add_column("markets", sa.Column("slug", sa.String(120), nullable=True))
    conn = op.get_bind()
    conn.execute(sa.text("UPDATE markets SET slug = lower(replace(title, ' ', '-'))"))
    with op.batch_alter_table("markets") as batch:
        batch.alter_column("slug", nullable=False)
        batch.create_unique_constraint("uq_markets_slug", ["slug"])
```

### Example 4 — CLI recipes

```bash
# generate new migration
alembic revision --autogenerate -m "add liquidity_usd"

# apply
alembic upgrade head

# roll back 1 step
alembic downgrade -1

# inspect current head of DB
alembic current

# show SQL without executing (review before upgrade)
alembic upgrade head --sql > /tmp/upgrade.sql
```

### Example 5 — Branch / merge workflow

```bash
# on branch A
alembic revision --autogenerate -m "branch a change"
# on branch B
alembic revision --autogenerate -m "branch b change"
# after merging code, reconcile:
alembic merge -m "merge a and b" <rev_a> <rev_b>
```

## 4. Gotchas / pitfalls

- **SQLite cannot `ALTER COLUMN`** — always wrap in `op.batch_alter_table(...)` / enable `render_as_batch=True` in `env.py`.
- **Autogenerate misses**: `CheckConstraint`, server defaults that don't match exactly, enum changes. Always review generated migrations.
- **Never edit a migration that's been applied to a shared DB.** Create a new one.
- **`target_metadata = Base.metadata`** must import the module defining all models, or autogenerate will miss tables.
- **Async env.py requires `async_engine_from_config` + `connection.run_sync(do_run_migrations)`** — plain sync pattern won't work with aiosqlite.
- **When upgrading SQLite → Postgres**: dump data, run fresh migrations on Postgres, restore via SQL. Alembic does not migrate *data* across dialects.
- **CI**: run `alembic upgrade head` before tests to ensure schema is current.
